# PRD Snapshot — 2026-06-11
Source: https://app.notion.com/p/82b66f3dc7ab466cb2aae8a4e924171a

> このページは、個人用AI投資アドバイザーの最新PRDです。NotionをSingle Source of Truthとし、Claude Codeはこのページを参照して実装します。

## 1. Product Vision
自分専用の投資ダッシュボードと、複数視点のAI投資委員会を構築する。
目的は、AIに売買判断を丸投げすることではなく、ポートフォリオの現状・リスク・目標とのギャップ・投資方針との整合性を、複数のエージェントが週次でレビューし、意思決定を支援すること。
あわせて、レポートは「意思決定支援」と「読み物として楽しめること」の両立を狙う。各エージェントは一貫した語り口を持ち、意見の対立そのものを読みどころとする。淡々としたアクション候補は別途構造化して提示し、忙しい週でもそこだけ見れば判断できるようにする。

## 2. Goals
- SBI証券を中心とした資産ポートフォリオをNotionに記録する
- SBI証券のCSVをNotionに定期アップロードし、スクリプトで取り込む
- SBI証券以外の資産は手動でNotion DBに追加・管理する
- 投資方針・目標・制約条件をNotionに記録する
- 複数のAIエージェントが異なる観点からポートフォリオを分析する
- 週1回程度、ポートフォリオ情報の更新・関連ニュース収集・分析を実施する
- 週次レポートをNotionに新規ページとして作成する
- Notionページを常に最新のPRDとして運用する
- Claude CodeはNotion PRDを参照し、実装時点のスナップショットをGitに保存してから実装する
- 4つの視点エージェントの分析を統合する「議長(Synthesizer)」の役割を設ける
- レポートは意思決定支援であると同時に、毎週読みたくなる読み物として設計する

## 3. Non-Goals
- 自動売買はしない
- 証券口座への発注はしない
- AIが投資判断を断定しない
- 「必ず上がる」「絶対買うべき」「確実に儲かる」などの表現は使わない
- 初期MVPでは、証券口座への直接ログインやスクレイピングは行わない
- 初期MVPでは、税務計算・配当履歴・取引履歴の完全対応は行わない
- 初期MVPでは、ニュース収集は必須ではなくP1扱いとする

## 4. Recommended Operating Model
Notionを最新仕様の正本とし、Gitには実装時点のPRDスナップショットを保存する。
```plain text
Notion PRD = 最新の正本
Git snapshot = その実装が参照した仕様
```
実装フロー：
```plain text
1. Claude CodeがNotion PRDを読み込む
2. docs/prd.snapshot.md に現在のPRDを保存する
3. PRD内の「決定事項」「P0」「P1」のみを実装対象にする
4. 「検討中」「将来アイデア」は実装しない
5. 実装する
6. テストする
7. 実装ログ・未対応項目をNotionに追記する
```

## 8. Notion Database Design

### 8.1 SBI CSV Import DB
SBI証券CSVをアップロードし、取り込みステータスを管理するDB。
| プロパティ | 型 | 内容 |
| --- | --- | --- |
| 名前 | title | 例：SBI保有資産CSV 2026-06-06 |
| CSVファイル | files | SBI証券からダウンロードしたCSV |
| 種別 | select | 保有資産 / 取引履歴 / 入出金 / 配当金 |
| 対象日 | date | CSVの基準日 |
| ステータス | status | 未処理 / 処理済み / エラー |
| 処理日時 | date | スクリプト実行日時 |
| エラー内容 | text | パース失敗・更新失敗など |
| インポート結果 | text | 作成件数・更新件数・スキップ件数 |
| メモ | text | 任意メモ |
処理ルール：
- スクリプトは `ステータス = 未処理` のCSVのみ対象とする
- 処理成功後は `処理済み` に更新する
- 処理失敗時は `エラー` に更新し、エラー内容を残す
- 同じCSVを二重処理しない
- CSVファイルはページ本文ではなくFilesプロパティに添付する

## 9. SBI CSV Import Specification

### 9.1 CSVの形式（実物確認済み）
本仕様は、実際にSBI証券からエクスポートしたCSV（「ポートフォリオ一覧」「一括表示」、Shift-JIS、総件数23件）を確認して確定した。
- 文字コードは **Shift-JIS（cp932）**。
- 改行は **LF と NEL（U+0085）が混在**する。パース時はNELも改行として扱う。
- 単一のフラットな表ではなく、**口座・商品区分ごとのセクションが縦に連結された複数セクション構成**。
- 先頭6行はメタ情報（"ポートフォリオ一覧" / "一括表示" / "PTS株価非表示" / "総件数：N件" / "選択範囲：…" / "ページ：N"）。
- 各セクションは「セクション見出し行 → 列見出し行 → 明細行（複数）」で構成され、続いて小計（「○○合計」見出し → 「評価額/含み損益/…」ヘッダ → 数値行）が入る。
- 末尾に「総合計」セクションがある。
- 米国株・現金残高・取引履歴・配当履歴はこのCSVには含まれない（別エクスポートが必要）。
確認できたセクション：
- 株式（現物/特定預り）
- 株式（現物/NISA預り（成長投資枠））
- 投資信託（金額/特定預り）
- 投資信託（金額/NISA預り（成長投資枠））
- 投資信託（金額/NISA預り（つみたて投資枠））
- 投資信託（金額/旧NISA預り）
- 国内債券（特定預り）

### 9.2 セクション→口座・資産クラスのマッピング
明細行に口座区分の列は無い。**口座と商品種別はセクション見出しから判定する。**
| セクション見出し | 資産クラス | 口座 |
| --- | --- | --- |
| 株式（現物/特定預り） | 日本株 | SBI特定 |
| 株式（現物/NISA預り（成長投資枠）） | 日本株 | SBI NISA成長投資枠 |
| 投資信託（金額/特定預り） | 投信 | SBI特定 |
| 投資信託（金額/NISA預り（成長投資枠）） | 投信 | SBI NISA成長投資枠 |
| 投資信託（金額/NISA預り（つみたて投資枠）） | 投信 | SBI NISAつみたて投資枠 |
| 投資信託（金額/旧NISA預り） | 投信 | SBI旧NISA |
| 国内債券（特定預り） | 債券 | SBI特定 |

### 9.3 株式・投資信託の列マッピング
株式・投信セクションは同じ10列構成（先頭セルのみ「銘柄（コード）」／「ファンド名」と名称が異なる）。すべての値はダブルクォート付きで、正の数値には先頭に `＋` が付く。
| CSV列 | 例 | Notionプロパティ | 備考 |
| --- | --- | --- | --- |
| 銘柄（コード）/ファンド名 | 8473 ＳＢＩ | 外部ID＋資産名 | 株式は「コード␣銘柄名」を最初の空白で分割。投信はコード無し |
| 買付日 | 2024/03/06 | （参考） | 「----/--/--」は複数約定の平均。null扱い |
| 数量 | 100 | 数量 | 投信は口数 |
| 取得単価 | 8315 | 平均取得単価 | 投信は1万口あたり |
| 現在値 | 7215 | 現在価格 | 投信は1万口あたり基準価額 |
| 前日比 | -523 | （参考） | MVPでは未使用可 |
| 前日比（％） | -6.76 | （参考） | MVPでは未使用可 |
| 損益 | +505200 | 評価損益 | ＋記号を除去 |
| 損益（％） | +78.84 | 評価損益率 | ＋記号を除去 |
| 評価額 | 721500 | 評価額 | 投信は小数あり |
取得額の算出（CSVに取得額列は無い）：
- 株式：取得額 = 取得単価 × 数量、評価額 = 現在値 × 数量
- 投信：取得額 = 取得単価 × 数量 ÷ 10000、評価額 = 現在値 × 数量 ÷ 10000（数量＝口数、単価＝1万口あたりのため）

### 9.4 国内債券の列マッピング
国内債券セクションは株式・投信と全く異なる列構成で、**各セルがダブルクォートで囲まれない**点にも注意する。
列：`銘柄, 利率(%), 償還日, 利払日, 買付日, 保有額面, 取得単価, 約定為替, 参考為替, 評価額`
例：`第１７８回利付国債（２０年）, 0.500, 2041/09/20, 03/09-20, 2026/01/16, 4150000, 72.48, --, --, 3007920`
- 銘柄コードは無い。取得単価は額面100あたりの単価。
- 取得額 = 保有額面 × 取得単価 ÷ 100。
- MVPでは「評価額」「取得額（算出）」「保有額面」のみHoldingsに反映し、利率・償還日・利払日・為替はメモまたは将来対応（P2）とする。

### 9.5 パース・スキップルール
- Shift-JIS（cp932）で読み込み、NEL（U+0085）も改行として扱う。
- 先頭メタ6行はスキップ。
- 「○○合計」「総合計」見出しと、その直後の「評価額/含み損益/…」ヘッダ行・数値行はスキップ（明細ではない）。
- 列見出し行（「銘柄（コード）」「ファンド名」「銘柄,利率(%)…」で始まる行）はスキップ。
- セクション見出し行を検出したら、現在の口座・資産クラスを切り替える。
- 数値正規化：先頭の `＋` を除去、カンマ除去、`--` や `----/--/--` や空文字は null。
- 株式の先頭セルは最初の空白で「コード」と「銘柄名」に分割（コードは `8473` や `186A`・`212A`・`147A` のようにグロース市場の英字付きコードもある）。
- 同一商品が複数口座にまたがる場合は別Holdingとして扱う（例：オルカンが特定/成長/つみたて/旧NISAに分散）。主キーは `データソース(SBI_CSV) + 口座 + 外部ID（または資産名）`。

### 9.6 エラー処理
- 必須列がない場合はエラーにする
- パースできない数値がある場合は該当行をスキップし、エラー内容に記録する
- 全体失敗時はCSV Import DBのステータスを `エラー` にする
- 部分成功時は作成・更新・スキップ件数を記録する

### 9.7 サンプルCSV管理
- 本物CSVをそのままGitに入れない
- 匿名化したサンプルCSVを `tests/fixtures/` に保存する
- 本物CSVはNotionのSBI CSV Import DBに添付する
- ローカルの本物CSV保存場所は `.gitignore` 対象にする

### 9.8 ニュース収集ソース（P1）
ニュース収集はP1として実装する。目的はニュースの網羅ではなく、週次レポートに使える「保有銘柄への含意」を拾うこととする。
初期実装では以下を採用する：
- 保有銘柄関連ニュース：Google News RSS
- 日本株の公式開示：TDnet
- 米国マクロ指標：FRED
初期実装では採用しない：
- CoinGecko：暗号資産をほぼ保有していないため、P0/P1初期実装からは除外する。将来、暗号資産の保有比率が上がった場合に再検討する。
段階導入候補：
- NewsAPI / GNews：Google News RSSの安定性や検索精度が不足した場合の代替・補完
- EDINET：日本株の有価証券報告書・四半期報告書・大量保有報告書などの深掘り
- SEC EDGAR：米国個別株を保有する場合の公式開示確認
運用ルール：
- 銘柄名・ティッカー・ファンド名・主要テーマを検索キーにする
- マクロニュースは独立章として扱わず、保有銘柄・資産クラスへの含意に翻訳してレポートに反映する
- ニュース記事の内容は投資判断の根拠の一部に留め、売買判断を断定しない

## 10. Agent Specification
エージェントの役割・重視指標・制約・プロンプト・初期エージェント一覧はすべて **Agent DB（§8.6）を正本として管理する**。Claude Codeはバッチ実行時にAgent DBの `有効 = true` の行を取得し、優先度順に実行すること。PRDにエージェントの個別定義は記載しない。
実装上の遵守事項（§13 Safety Rules と合わせて適用）：
- すべてのエージェント出力に根拠・不確実性・反対意見を含める
- 投資判断を断定しない。最終判断はユーザーが行う前提で書く
- 議長（Synthesizer）は他4体の出力を入力とした2段階目で実行する
- モデル割り当て：Provider / Modelは設定で切り替え可能にする。初期候補はGeminiとし、個別ペルソナ4体は低〜中コストの高性能モデル、議長は高品質モデルを使う（§17 Decisions参照）

## 11. Weekly Report Specification
週次レポートはNotionページとして作成する。
セクション0・6・8は議長(Synthesizer)が生成する。マクロは独立章にせず、保有銘柄への含意に翻訳した形でのみ登場させる（マクロの一次情報は別途デイリーで確認する前提）。
推奨構成：
```plain text

## 0. 振り返り（答え合わせ）  ※P1
- 「1週間前」と「1か月前」の2段階で過去レポートを振り返る
- 各時点の推奨が、その後どうなったか
- 各エージェント／議長の自己採点

## 1. 今週の総括

- 前週比
- 主な変動要因

## 2. ポートフォリオ状況
- 資産クラス別
- 通貨別
- 地域別
- リスク分類別

## 3. FIRE目標との差分
- 目標資産額
- 現在位置

## 4. 保有銘柄関連ニュース
- 保有銘柄に直接関係するニュース中心
- マクロ（金利・為替・セクター）は「保有銘柄への含意」に翻訳した形でのみ記載

## 5. エージェント別コメント

### 老後安全性重視エージェント
### 投機チャンス探索エージェント

## 7. 今週の確認事項
```

## 12. Execution Flow
週次バッチの流れ：
```plain text
1. Notion PRDを読み込む
2. 必要なDBスキーマ・設定を読み込む
3. 未処理のSBI CSVを探す
4. CSVファイルをダウンロードする
5. CSVをパースする
6. Holdings DBを更新する
7. 手動資産を含めてポートフォリオ全体を集計する
8. Portfolio Snapshotを作成する
9. 関連ニュースを収集する（P1。初期ソースはGoogle News RSS / TDnet / FRED）
10. 1週間前・1か月前のAgent Analysis Results / 議長まとめを読み込む（答え合わせ用・2段階・P1）
11. 有効なエージェント一覧を取得する
12. 各エージェントが分析する
13. 議長が4視点を統合し、対立・総合評価・アクション候補を生成する
14. 分析結果をAgent Analysis Results DBに保存する
15. 週次レポートを生成する（冒頭に答え合わせを含める）
16. Weekly Reports DB / Pageに保存する
17. 実行ログを残す
```

## 13. Safety Rules
- 自動売買はしない
- 証券口座への発注はしない
- 投資判断を断定しない
- 必ず根拠・反対意見・不確実性を含める
- 投機提案には、最大損失・ポジションサイズ・損切り条件・期間を含める
- レポートは助言ではなく意思決定支援として扱う
- 最終判断はユーザーが行う
- APIキー・Notionトークン・Claude APIキーは `.env` で管理し、Gitに含めない
- 本物のCSVや個人情報をGitに含めない

### 14.1 Claude Code向けドキュメント方針
NotionをSingle Source of Truthとする原則を崩さないため、リポジトリ内ドキュメントは仕様を二重管理せず、次の役割分担とする。
- `CLAUDE.md`：Claude Codeが起動時に自動ロードする恒久ルール（手動資産の上書き禁止・投資判断の断定禁止・秘密情報や本物CSVのコミット禁止）と、「正本のPRDを読み込め」という道標。仕様そのものはここに書かない。
- `docs/prd.snapshot.md`：その実装が参照したPRDの凍結コピー（§4の運用に基づく自動生成）。
- `docs/sbi_csv_spec.md`：§9 SBI CSV仕様を実装・テスト時に参照しやすくするための補助スナップショット（正本はあくまでNotion）。エージェント仕様（§10）はAgent DBが正本のため、リポジトリには置かない。
- `.claude/commands/`：`/weekly-run`・`/import-csv`・`/snapshot-prd` などの定型作業をカスタムスラッシュコマンド化する。

## 17. Decisions
- Notion PRDを最新の正本とする
- AI実行形態は外部LLM API / SDK直叩きとし、Provider / Modelは設定で切り替え可能にする
- 初期実装ではGemini APIを第一候補とする。Claude / OpenAI / OSS系API等は比較・代替候補として扱う
- 本番週次バッチでは、個別ペルソナ4体は低〜中コストの高性能モデル、議長(Synthesizer)は高品質モデルを使う。初期候補はGemini系モデルとする
- 議長(Synthesizer)はレポート全体の品質を左右するため、コストよりも統合力・日本語の読みやすさ・断定しすぎない表現を優先してモデルを選ぶ
- モデル品質検証として、必要に応じてGemini版 / Claude版などのサンプル週次レポートを比較し、読み物としての面白さ・安全性・根拠提示・コストを評価する
- コスト管理のため、Provider・モデル・最大トークン数・実行回数・概算利用料を実行ログに記録する
- 実装時にはPRDスナップショットをGitに保存する
- SBI証券CSVはNotionのFilesプロパティに添付して取り込む
- SBI以外の資産は手動入力で管理する
- 初期MVPでは自動売買をしない
- 初期MVPは「4つの視点エージェント（FIRE／老後／投機／リスク管理）＋1つの統合・議長役」で構成する
- 議長(Synthesizer)は独立したエージェント（Agent DBに1行）として実装する。ただし実行は2段階目とし、4体の出力を入力として最後に走らせる（1〜4を回す→集約→議長を回す）
- 手動資産は手動更新とする（自動の時価取得はしない。各種比率・総資産額の陳腐化は許容する）
- SBI CSVはShift-JIS・複数セクション構成として取り込み、口座区分はセクション見出しから判定する（§9）
- 取得額はCSVに列が無いため、株式＝取得単価×数量、投信＝取得単価×数量÷10000で算出する（§9.3）
- Holdingsの口座区分に「SBI旧NISA」を追加し、旧NISA預りは独立した口座として管理する（§8.3）
- 国内債券はMVPでは評価額・取得額（算出）・保有額面のみHoldingsに反映し、利率・償還日・利払日・為替はP2とする（§9.4）
- 週次レポートをNotionページとして作成する
- 取り込むCSVは保有資産（ポートフォリオ一覧）のみとし、取引履歴CSVは早期には取り込まない（§9）
- 手動資産はManual Assets DBを別に作らず、Holdings DB内（`データソース = Manual`）で管理する（§8.3）
- 答え合わせ（過去レポート参照）を実装し、「1週間前」「1か月前」の2段階でさかのぼる（§11 / §12）
- 「読み物として楽しめること」を正式ゴールに昇格させる。安全ルール（断定禁止等）は維持しつつ、最終判断は読者が行う前提のもとで読み物としての面白さを重視してよい
- 投機枠の上限は総資産の5%とする（§10.4）
- エージェント別モデルは固定せず、Provider / Modelを設定で切り替え可能にする。初期候補はGeminiとし、個別ペルソナ4体は低〜中コストの高性能モデル、議長(Synthesizer)は高品質モデルを使う
- ニュース収集の初期ソースは、保有銘柄関連ニュースをGoogle News RSS、日本株の公式開示をTDnet、米国マクロ指標をFREDとする（§9.8）
- CoinGeckoは暗号資産をほぼ保有していないため、P0/P1初期実装から除外する。暗号資産の保有比率が上がった場合に再検討する（§9.8）
- NewsAPI / GNews / EDINET / SEC EDGARは、初期実装後の段階導入候補とする（§9.8）
- 週次実行日はPRDで固定せず、サーバーへのcron登録時に決める
